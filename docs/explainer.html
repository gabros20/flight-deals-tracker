✈️ flight-deals — how it works

A deal-finding engine for Ryanair & Wizz Air trips from Budapest, built to be driven by an AI assistant over Telegram. The strategy lives inside the tool; the agent just picks a verb.

The big picture

One command surface, four layers

You, a cron job, and Hermes all run the exact same commands. Every layer below the top is deterministic — same input, same procedure — which is what makes price history, alerting, and agent behavior trustworthy.

📱 You, via Telegram human intent

“Find me a seaside getaway Aug 22–24, under €200” · “any flight news?” · “watch Corfu under €150”

↓

🤖 Hermes / Claude + SKILL.md router translation only

A ~77-line intent table: utterance → one exact command. Weak models don’t plan pipelines — they follow. Strong agents may drop to the spec layer. A drift-pin test fails CI if the skill ever documents a flag the CLI lost.

↓

⌨️ CLI intent verbs pipelines live here

getaway · oneway · check <deal_id> · watch add/list/rm · brief --send · where list/show — plus the power layer: plan --spec (cost preview, zero network) and run --spec.

↓

🧠 Engine spec → planner → combine → confirm

Compiles a declarative search spec into a typed call plan, executes it under a global ~1 req/s rate limiter, pairs fares, dedupes (exact beats approximate), ranks, and confirms estimates before you see them.

↓

🌐 Free data plane no paid APIs

Ryanair farfnd — true paired round-trip fares, month calendars, “anywhere” sweeps (exact) · Wizz timetable — daily minima both directions, HUF→EUR converted, API version auto-rediscovered on drift (approximate ±10%).

A search, end to end

“Seaside, Italy or Spain, Aug 22–24, 5–8 nights, under €200”

$ flight-deals getaway --depart 2026-08-22..2026-08-24 \
    --where "seaside | italy | spain" --nights 5-8 --budget 200

Resolve the category

--where is a tag algebra (& | ! ()) over 76 curated airports. "seaside & (italy | spain)" → 23 airports. Typos get a did-you-mean and cost zero API calls.

Compile a call plan — before any network

One Ryanair anywhere call covers every destination’s cheapest pair in the window; Wizz needs one timetable call per candidate route. plan --spec shows the exact calls + estimated seconds; a cap (default 40 calls) refuses runaway sweeps with a corrected command.

Execute, pair, merge

Ryanair pairs are exact. Wizz day-fares are paired into round-trips within your nights window — estimates. On a price tie, exact beats approximate, always.

Estimate → confirm

Any estimate that would cross your budget or make the top list gets a fresh exact-date re-query first (including a margin band, so a deal estimated 10% over budget whose true price is under still gets rescued). A failed confirm never invents a price.

Rank, explain, snapshot

Deals get a stable deal_id, a why-string (“€113 vs typical €160, 29% below, 41 observations”), and an append-only snapshot so check <deal_id> works days later.

Answer in one honest envelope

{
  "results":  [ {deal_id, price_eur, price_confidence, legs, why, links…} ],
  "summary":  "10 deals: cheapest CFU €113 rt Aug 22–27 (exact)…",
  "sources":  { "ryanair": "ok", "wizzair": "error" },  // failures are visible, never hidden
  "next":     [ "flight-deals check 24affe56a9" ]      // exactly ONE suggested move
}

Exit codes: 0 ok (even empty — “no deals” is an answer) · 1 provider down · 2 your input, with the corrected command in hint.

Trip shapes

Four ways to build a cheap trip

Enumerate wide on cheap day-level data, verify the shortlist with exact queries. Ground hops are priced in (marked ~ — they’re curated estimates) and every composite total is what ranking and budget act on.

S1 / S2 · direct
BUD✈CFU✈BUD

The core: true paired round-trip fares, one-way sweeps. Default shape.

S3 · extended origin
BUD🚌VIE✈NAP✈VIE🚌BUD

Vienna & Bratislava have huge low-cost bases 2–3h from Budapest. Same sweep from there, +2× bus cost (~€42 VIE). Shown only when it genuinely beats direct.

S4 · open-jaw
BUD✈BRI🚆NAP✈BUD

Fly into one city, train across, fly home from another (both directions evaluated, cheaper wins). Two exact one-way fares + ground — two separate tickets, combined locally. Live find: €114.68.

S5 · via-hub — not yet
BUD✈HUB✈?

Same-day self-transfers need flight-time verification. The spec accepts it; the planner politely refuses until it’s built right.

$ flight-deals getaway … --shapes direct,extended-origin,open-jaw   # opt-in
The monitoring loop

Watch → brief → exactly one alert

A watch is a saved search — a YAML file with a spec, a schedule, and a threshold:

# data/searches/corfu-august.yaml
spec:     { where/route, depart, nights }
schedule: "daily 08:30"
alert:    { max_price: 150, notify: telegram }
agent_prompt: # optional — agentic review

launchd runs brief --send 3×/day: due searches execute, results append to price history, drops cross the alert machine, past-dated state gets pruned, and one chunked HTML digest lands on your phone.

Alert state machine (per route + month)

new— confirmed price < threshold → alerted→ suppressed— ≥15% further drop, or expiry → re-armed

The 15% re-alert bar sits above Wizz’s ±10% noise, so estimates can’t make it flap — and only confirmed-exact prices may alert, ever. Alerts persist as pending before the send and are marked delivered only on Telegram’s confirmation: a transient outage re-sends next run instead of silently losing your deal.

Trust model

The tool is not allowed to lie

SignalMeaning
exactRyanair’s own bookable fare, or a Wizz price re-verified on the exact date. Only these can trigger alerts or final budget decisions.
approximateWizz timetable cache (±10%) or a day-level pairing — always labeled, confirmed before it matters, never alert-worthy on its own.
~€42 groundBus/train estimates are curated, marked with ~, and included in the ranked total — never hidden.
sources: errorA provider failing is reported in every response and in the digest (“results may be incomplete”). “No deals found” always means we looked.
